## 2. 추천안 + 이유

### 추천: **포트원(PortOne)을 통한 본인인증 연동** — 1차로 **통합 본인인증(KG이니시스)**, 최소안으로 **휴대폰 본인인증(다날)**

**이유 (BOHUMFIT 맥락 근거):**

1. **단일 연동으로 결제+본인인증을 함께 관리** — 포트원은 결제(PG)와 본인인증을 한 SDK/콘솔로 묶어 제공한다. BOHUMFIT는 이미 정기결제를 쓰므로, 결제·인증을 한 곳에서 운영하면 키·정산·CS·장애 대응이 단순해진다.
2. **초기 규모(월 수십~수백 건)에 적합** — 본인인증은 보통 **건당 과금** 구조라 초기 물량에선 직계약의 행정 부담보다 애그리게이터(포트원) 경유가 총비용·운영 측면에서 합리적. (단가는 견적 확정 필요 — 1번 표 참조)
3. **CI 확보 가능** — 통합 본인인증/휴대폰 본인인증 모두 본인확인기관 경유로 **CI를 응답**받을 수 있어, 번호가 아니라 CI로 **진짜 1인 1계정**을 구현할 수 있다(카카오 경유 시에도 CI 제공, 단 KG이니시스 별도 서류).
4. **민감정보(병력) 취급 서비스의 신뢰** — 실명·실본인 확인은 어뷰징/대리조회를 줄이고, 개인정보 분쟁 시 "본인확인을 거친 계정"이라는 방어선을 제공한다.

**대안(생태계 일원화 관점):** 이미 토스 빌링을 쓰므로 **토스페이먼츠 본인인증**으로 키·정산을 토스로 통일하는 선택도 합리적이다. 다만 **지원 수단·과금·계약 조건을 이번 조사에서 확정하지 못함(확인 불가)** → 토스 영업/공식 문서로 견적·기능을 확인한 뒤 포트원안과 **비용·기능 직접 비교** 후 최종 결정 권장.

> ※ 본 추천은 공개 정보 기반의 설계 제안이며, **최종 선택은 두 곳(포트원·토스) 실제 견적과 계약 조건을 받은 뒤** Human이 결정해야 한다.

## 3. 연동 아키텍처 설계 (코드 아님 — 흐름·교체 지점만)

### 3.1 인증 흐름

```
[프런트] PhoneVerify 화면
   │  ① 본인인증 SDK 호출(포트원 SDK: 인증창 오픈 → 사용자 PASS/간편인증)
   ▼
[인증사/PG] 본인확인 수행 → 결과 토큰(impUid 등) 프런트로 반환
   │  ② 프런트가 결과 토큰 + Supabase access_token 을 백엔드로 전달
   ▼
[백엔드] POST /auth/verify-phone (실연동 버전)
   │  ③ 서버가 포트원 REST API로 결과 토큰 검증(서버-서버) → 인증 원본(이름·생년월일·전화·CI) 수신
   │  ④ CI 해시(예: SHA-256(CI + 서버 비밀 salt)) 계산
   │  ⑤ profiles 중복 검사: 같은 ci_hash 가 다른 user 에 있으면 거부(1인 1계정)
   │  ⑥ profiles upsert: phone, phone_verified=true, ci_hash 저장 (CI 원문은 저장하지 않음)
   ▼
[프런트] 성공 → 원래 경로 복귀(086 from 흐름 그대로) / 중복 → 한국어 안내
```

### 3.2 CI 기반 1인 1계정

- **번호가 아니라 CI로 동일인 판별.** 번호는 변경·재발급·타인 명의로 우회 가능하지만, CI는 개인 고유·불변값이라 어뷰징 방어가 강하다.
- `profiles`에 **`ci_hash` 컬럼**(텍스트)을 두고 **CI 원문 대신 해시(salt 포함)만 저장**. 중복 판별은 `ci_hash` UNIQUE(부분 인덱스: `phone_verified=true`)로.
- 088의 `phone` 부분 UNIQUE는 **임시방편**(번호 재사용만 차단). 087 실연동 후 **`ci_hash` UNIQUE가 진짜 1인 1계정**을 담당하고, `phone` UNIQUE는 보조로 유지/완화.

### 3.3 기존 스텁 → 실연동 교체 지점

| 위치 | 현재(스텁) | 087 실연동 후 |
|---|---|---|
| `src/pages/PhoneVerify.tsx` | 번호 직접 입력 → POST `/auth/verify-phone` | 본인인증 **SDK 호출**로 교체, 입력창 제거. 결과 토큰을 백엔드로 전달 |
| `backend/main.py` `verify-phone` | 받은 phone으로 `profiles.upsert(phone_verified=true)` | **결과 토큰 서버 검증 → CI 수신 → ci_hash 중복검사 → upsert(phone, phone_verified, ci_hash)** |
| `supabase/migrations` | phone 컬럼·부분 UNIQUE(088) | `ci_hash` 컬럼 + `ci_hash` 부분 UNIQUE 인덱스 추가(신규 마이그레이션) |
| 환경변수 | 없음 | 포트원/토스 **API 키·시크릿**(서버), 가맹점 식별자 등 — Railway/Vercel 분리 |

## 4. 개인정보 처리

- **수집 최소화:** 본인확인으로 들어오는 항목(이름·생년월일·전화·성별·내외국인·CI/DI) 중 **서비스에 꼭 필요한 것만** 저장. 1인 1계정 목적이면 **`ci_hash`(해시)와 `phone`** 정도로 충분. **CI 원문·주민번호는 저장하지 않음.**
- **암호화/보관:** `ci_hash`는 서버 비밀 salt를 섞어 단방향 해시. 전송 구간 TLS. CI 원문은 검증 직후 폐기(미저장). 보관 최소기간·파기 정책 수립.
- **처리방침 반영 필요사항:** 본인확인 수집 항목·목적·보유기간·**처리 위탁(본인확인기관/PG)** 고지, 동의 절차. 개인정보처리방침/이용약관 개정 필요(법무 검토 — Human 결정 사안).
- **법적 근거:** 정보통신망법상 본인확인은 **지정된 본인확인기관** 경유가 원칙. 자체 구현이 아니라 PG/대행을 통한 적법 경로 사용.

> ⚠️ 개인정보·법무·약관은 **Human 결정 사안**. 본 문서는 기술 설계 관점의 요건만 정리하며, 실제 약관/처리방침 문구는 법무 검토 후 확정해야 한다.

- **토스페이먼츠 본인인증의 지원 수단·과금·계약 조건** — 공식 문서 클라이언트 렌더링으로 본문 미취득 → Human/Claude(브라우저) 추가 조사 필요.
- **다날·KG이니시스 직계약 조건·물량 할인** — 영업 문의 필요.
- **연동 난이도 정량치**(개발 공수) — 실제 SDK·문서 정독 후 산정 필요.
