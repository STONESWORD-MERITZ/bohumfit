# BOHUMFIT-087: 본인확인(휴대폰/PASS) 연동 설계 조사 — 문서만, 코드 연동 없음
## 목적
설계사용 민감정보(병력) 서비스에 적합한 실제 본인확인 연동 방식을 조사해 설계 문서를 작성한다. 실제 키·계약·서버 연동은 사업자 승인 후 별도 태스크. 이번엔 결정 가능한 설계안까지만.
## 배경
- 현재 폰인증은 074 스텁(통신사 검증 없음). 같은 번호로 카카오·구글 다계정 통과 문제 확인됨.
- 결제는 토스페이먼츠 사용 중(빌링). 사업자 등록 신청 완료, 승인 대기.
## 산출물
- docs/identity-verification-plan.md (신규) 1개. 아래 목차로 작성.
## 작성 항목
1. 후보 비교: 포트원(PortOne) 통합본인인증 / 토스페이먼츠 본인인증 / 다날 직계약 / KG이니시스 통합인증
   - 각 항목: 지원 수단(PASS·카카오·네이버·토스 등), 과금 구조(정액/건당), 가입비, 연동 난이도, 계약 선행조건
   - ★ 단가·정책은 변동 가능 → "조사 시점 추정치, 실제는 견적 필요"로 명시. 추측을 확정처럼 쓰지 말 것.
2. 추천안 1개 + 이유: BOHUMFIT 규모(초기 구독 수십~수백 건/월)·기존 토스 사용·민감정보 취급을 근거로.
3. 연동 아키텍처 설계(코드 아님, 설계만):
   - 프런트 SDK 호출 → 백엔드 검증 콜백 → CI(연계정보) 수신 → profiles 저장 흐름
   - ★ CI 기반 1인 1계정: 번호가 아니라 CI(개인 고유 불변값)로 동일인 판별. profiles에 ci 컬럼(해시 저장) 두는 안.
   - 기존 스텁(PhoneVerify/verify-phone)을 실연동으로 교체할 지점 표시
4. 개인정보 처리: 수집 항목(이름·생년월일·전화·CI) 최소화, 보관·암호화, 처리방침 반영 필요사항.
5. 선행조건 체크리스트: 사업자 승인 / 대행사 계약 / 키 발급 / 처리방침 갱신 / 키 환경변수(.env, Railway, Vercel) 분리.
6. 다음 구현 태스크 분해 제안(승인 후 진행할 BOHUMFIT-xxx 후보 목록).
## 방법
- 토스페이먼츠 본인인증 공식 문서, 포트원 공식 문서 등 1차 출처 위주로 조사. 출처 URL을 문서에 남긴다.
